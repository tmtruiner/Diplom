# Продуктовый ориентир интерфейса

Дата обзора: 15 июня 2026 года.

Рассмотренные продукты: Gainsight, Totango, ChurnZero и Planhat.

## Общая модель продукта

Зрелые системы управления удержанием клиентов строят сценарий в три уровня:

1. Обзор клиентской базы: показатели, распределения, динамика и проблемные категории.
2. Рабочий список: фильтры, группировка, сортировка, массовые действия и выгрузка.
3. Карточка клиента: объяснение состояния, история изменений и рекомендуемое действие.

Главный пользовательский путь:

`обнаружить риск -> открыть выборку -> понять причину -> выполнить действие`

## Что используют продукты

### Gainsight

- Сводная оценка состояния и несколько составляющих с отдельными весами.
- Разделение причин по группам: вовлечённость, использование, поддержка, результат.
- История изменения оценки.
- Карточка клиента как единый центр контекста и действий.
- Рабочие задачи и предупреждения рядом с аналитикой.

### Totango

- Сохранённые категории клиентов с фильтрами по состоянию и стоимости договора.
- Цветовые уровни состояния вместо перегруженной числовой аналитики.
- Краткая сводка над таблицей: количество клиентов, состояние и сумма договоров.
- Группировка, настройка столбцов и быстрые действия с выборкой.
- Готовые сценарии работы для разных состояний клиента.

### ChurnZero

- Оценка состояния, вероятность ухода и динамика во времени.
- Предупреждения об изменении состояния.
- Составная оценка из поведения, использования продукта, поддержки и обратной связи.
- Анализ взаимодействий и тональности общения.
- Рекомендации и автоматические действия при достижении порогов риска.

### Planhat

- Плотная таблица клиентского портфеля с компактными цветными индикаторами.
- Боковая панель выбранного клиента без ухода со страницы списка.
- Настраиваемые поля, показатели и временные графики.
- Фильтры, категории, владельцы клиентов и финансовые показатели.
- Явное разделение обзорной таблицы и подробного контекста.

## Визуальные закономерности

- Светлый нейтральный фон и белые рабочие поверхности.
- Один основной цвет для навигации и действий.
- Красный, жёлтый и зелёный применяются только для состояния и риска.
- Цвет сопровождается текстом или значком, а не является единственным сигналом.
- В таблицах используются короткие индикаторы, а объяснение переносится в карточку.
- Выбранная строка выделяется деликатно: фоном или цветной полосой.
- В подробной карточке сначала показывается состояние, затем причины и действие.
- Динамика важнее одиночного числа: рядом с оценкой показывается изменение за период.
- Числа сопровождаются контекстом: периодом, единицами и причиной изменения.

## Правила для Churn Analytics

### Таблица клиентов

Таблица отвечает на вопрос «с кем работать» и содержит:

- идентификатор клиента;
- вероятность ухода;
- группу риска;
- категорию клиента;
- оценочные расходы;
- рекомендуемое действие.

Для таблицы нужны:

- закреплённый заголовок;
- сортировка по ключевым столбцам;
- сохранение текущих фильтров;
- количество найденных клиентов;
- выгрузка текущей выборки;
- компактное выделение выбранной строки.

### Правая карточка

Карточка отвечает на вопросы «почему» и «что делать»:

1. Оценка состояния клиента и словесный статус.
2. Дата последнего прогноза.
3. Разложение оценки по причинам со штрафами.
4. Основной фактор риска с коротким объяснением.
5. Рекомендуемое действие, его причина и приоритет.
6. Выручка под риском как контекст важности.

Не следует повторять в карточке отдельным блоком вероятность, группу риска,
категорию и рекомендацию, если они уже видны в выбранной строке таблицы.

### Аналитическая панель

- Любая карточка или часть графика должна открывать соответствующую выборку.
- Над графиками показываются ключевые показатели и период данных.
- Для состояния используются одинаковые пороги и цвета на всех страницах.
- Желательно показывать изменение относительно прошлого прогноза.

## Терминология интерфейса

- `health score` -> `оценка состояния клиента`;
- `scoring` -> `прогноз` или `расчёт прогноза`;
- `dashboard` -> `аналитическая панель` или `обзор`;
- `drill-down` -> `переход к деталям`;
- `pipeline` -> `процесс обработки данных`;
- `dataset` -> `набор данных`;
- `segment` -> `категория клиентов`, если это не техническая документация;
- `playbook` -> `сценарий работы`;
- `alert` -> `предупреждение`;
- `customer success manager` -> `специалист по работе с клиентами`.

## Приоритет развития

1. Переработать правую карточку: состав оценки вместо повторения строки.
2. Добавить динамику оценки состояния между прогнозами.
3. Добавить предупреждения о резком ухудшении.
4. Добавить статус обработки рекомендации.
5. Добавить сохранённые наборы фильтров.
6. Добавить ответственного специалиста, если появится соответствующее поле.
